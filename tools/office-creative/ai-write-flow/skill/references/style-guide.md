# style-guide.md — 写作风格规范（执行约束层）

> 所有条目均为硬约束，不受 persona.md 内容影响。冲突时本文件优先。

---

## 第一人称（强制）

全文统一使用'我'的视角叙述。无论文章类型，强制执行，不可绕过。

---

## 句子长度

- 参考范围：15-25 字
- 语义完整性优先，不因达到字数上限而强制截断
- 单句超过 35 字时，在逗号处优先断开

---

## 段落规则

- 无硬字数上限，完整性优先
- 手机端参考：单段不超过 5 行
- 每段 1-2 个主要观点，多则拆段

---

## 加粗策略

每段话有且仅有一个核心关键词加粗，形成扫读路径。

---

## 口语化推荐词汇

以下为推荐自然融入的词汇，不强制硬插：

- 语气副词：确实、真的、说实话、很明显
- 情感词：香、爽、坑、牛
- 语气词：啊、呢、吧、嘛

---

## 数据化表达（强制）

用具体数字替代模糊描述：

- ❌ '启动时间显著提升'
- ✅ '启动时间从 6.5 秒降到 2.1 秒，快了 65%'

---

## 禁用表达清单

### AI 腔套话（强制删除）

- 在当今时代 / 在当今 AI 技术飞速发展的时代
- 综上所述 / 总的来说 / 归根结底
- 值得注意的是 / 不难发现 / 显而易见 / 毫无疑问 / 可以说
- 通过…可以 / 有助于 / 使得 / 从而实现
- 首先…其次…最后（机械三段）
- 不是…而是… / 虽然…但是… / 一方面…另一方面

### 书面词汇（强制替换）

| 禁用 | 替换 |
|------|------|
| 显著提升 | 具体数字 |
| 充分利用 | 用好 |
| 进行操作 | 直接用动词 |
| 获取结果 | 得到结果 |
| 实现功能 | 具体功能名称 |
| 相关方案 | 具体方案名 |

### 空洞表达（强制替换）

| 禁用 | 替换 |
|------|------|
| 具有显著优势 | 具体数据/事实 |
| 效果非常好 | 具体感受+数据 |
| 值得推荐使用 | 直接说推荐原因 |

### 夸张美学（强制替换为数据）

- 充满活力 / 蓬勃发展 / 里程碑 / 革命性 / 重塑
- 替换原则：形容词 → 具体数值或事实描述

### 模糊归因（强制补充来源）

- 专家认为 / 研究表明 / 据统计 / 众所周知
- 替换原则：必须指明具体机构、作者或论文链接

### 助手感开场白（强制删除）

- 当然可以 / 很高兴为您 / 没问题 / 作为一名 AI

## 维度对比表（强制）

针对工具对比、功能更新、重构评测，**必须**引入以下格式的 Markdown 表格：

```markdown
| 维度 | 以前 (Before) | 现在 (Now) |
|------|-------------|-----------|
| 启动时间 | 5.2 秒 | 1.8 秒 |
```

## Markdown 标题层级规范（强制）

### 三层结构定义

| 层级 | 用途 | 规则 |
|------|------|------|
| `#` | 文章标题 | 全文唯一，仅出现一次 |
| `##` | 一级章节标题 | 未来章节配图的候选单位 |
| `#### ①②③` | 章节内扫读小标题 | 每个 `##` 内独立编号，从 `①` 开始，后接对应正文段落 |

禁止使用 `###` 层级。

### 标准章节块结构

```markdown
# 文章标题

开头段落（此处不允许出现 ####）

## 一级章节标题

#### ① 小标题
正文段落...
#### ② 小标题
正文段落...
#### ③ 小标题
正文段落...

## 另一个章节

#### ① 小标题
正文段落...
#### ② 小标题
正文段落...

## 写在最后

结尾段落（不允许 ####）
```

### 规则细则

- **`#` 唯一性**：全文有且仅有一个 `#` 标题（文章总标题）
- **禁止 `###`**：不允许使用三级标题，层级只有 `#` → `##` → `####`
- **开头段落**：`#` 后、第一个 `##` 前的开头段落中不允许出现 `####`
- **普通 `##` 章节**：每个普通 `##` 章节建议包含 2-3 个 `####` 小标题
- **`####` 编号规则**：每个 `##` 内的 `####` 编号独立，从 `①` 开始，按 `①→②→③` 连续递增，不可跳号
- **配图候选单位**：`##` 章节块（标题 + #### 小标题组 + 正文摘要）是未来章节配图的语义输入单位，不是每段、也不是每个 `####`
- **固定结尾**：文章必须以 `## 写在最后` 作为最后一个 `##` 章节（唯一出现一次），该章节不参与配图候选，**不得包含 `####`**
- **正文位置**：普通章节内按 `#### 小标题` → 正文段落交替展开，不得把所有 `####` 集中堆在章节开头

## 文章结构模板

### Type 1：实战评测

逻辑链：Hook（痛点）→ Effect（直接成果）→ Method（实现过程）→ Pitfalls（踩坑）→ Conclusion

要点：
- 前 10% 篇幅展示最终效果或性能对比表
- 背景介绍不超过 200 字
- 至少 2 个真实踩坑案例
- 结尾必须包含'5 分钟上手清单'

### Type 2：深度解析

逻辑链：Hook（反常识/戏剧性）→ Visual（架构图）→ Reveal（3 层拆解）→ Insight（金句）

要点：
- 开篇通过反常识或故事性抓住读者
- 技术原理必须用生活化比喻
- 每个小结结尾有一句可转发的金句

### Type 3：教程指南

逻辑链：Introduction → Prerequisite → Step-by-Step → Troubleshooting → Tip

要点：
- 所有操作标 Step 1, Step 2 序号
- 关键配置给全量代码，不给片段
- 每步操作后说明'你应该看到什么输出'

### Type 4：经验分享

逻辑链：背景 → 遇到问题 → 解决过程 → 经验总结

要点：
- 诚恳分享，避坑指南为主
- 所有案例来自真实经历，数据可验证
- 语调：朋友间聊天，不是讲课

## 开头三种写法

**经历引入（最常用）：**
`[真实经历片段] + [问题描述] + [结果感受]`

**问题痛点引入：**
`[当前普遍存在的问题] + [具体场景] + [引入解决方案]`

**数据震撼引入：**
`[具体数字] + [对比说明] + [背后故事]`
