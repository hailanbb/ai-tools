A **broad, mostly-accurate health audit** — its core verdict ("production-ready, no critical defects, concurrency class closed") matches reality, and it correctly confirms the security posture. **But it is materially STALE**: it explicitly states "the full rc.24 audit report remains valid" and **carried over four findings that rc.28 already closed**, plus repeated the prior round's count errors. Two findings are genuinely actionable (one LOW, one MEDIUM). Net: useful as a 2nd-perspective confirmation of health, NOT reliable on its finding list — exactly why each item must be re-verified, not actioned on faith.

## Rejected / not-actioned (documented)

- **M-3** (22 bare catch) — miscounted (49, zero empty), all deliberate fail-soft; same rejection as rc.24 round.
- **M-2** TF-IDF `WeakMap` "unbounded at 100K notes" — **investigated, not a leak**: `tfidfCache` is a `WeakMap<Vault, …>` (search.ts:336) — one entry per live `Vault` object, GC'd when the vault is. It holds ONE cache per vault (overwritten on rebuild via `.set`), not linear growth. The auditor's "grows linearly with unique vault contents" misreads it. No fix. (A genuine bound already exists: it's keyed by object identity, not content.)
- **M-1** structured logger, **M-4/M-5** god-function refactors, **M-6** watcher throttle, **M-7** hnsw-m flags, **M-10** release environment, **L-1/L-4/L-5/L-6/L-8/L-9/L-10** — all valid-but-non-urgent hardening/DX; deferred (documented), no correctness impact.

## Lesson

This round reinforces the rc.24 round's lesson from the opposite direction: a competent external auditor can be **stale** (carrying prior findings without re-checking fixes) as easily as it can **miss** a live CRITICAL. Four of its findings were closed 4 RCs before the audit ran. The v3.6.1 ≥2-independent-auditors-with-different-methodologies gate stands: a single auditor's finding list is not trustworthy without per-item re-verification against current code.
