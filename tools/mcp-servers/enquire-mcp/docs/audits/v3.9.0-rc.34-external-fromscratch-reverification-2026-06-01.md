## The HIGH finding — REJECTED (auditor methodology error)

**H-DOC-1 "test count overclaim: docs say 1024, runtime is 1088 (1086+2 skipped); 64-test gap unidentified."** → **Not a defect.**
- The canonical, deliberate metric is **source `it()` count = 1024** (verified exact). Docs claim 1024 *because that is the count by design*; `docs-consistency.test.ts` pins tool/prompt counts to registered reality but the test-count figure is intentionally the source-`it()` number.
- The auditor's "no `.each`/`forEach`/dynamic `it()` found to explain the gap" is **false on inspection**: the gap is data-driven loops — e.g. `tests/redos-guard.test.ts` has `for (const p of catastrophic) { it(...) }` and `for (const p of safe) { it(...) }` over ~30-entry arrays (one source `it(`, many runtime cases), plus `for`-loop-wrapped `it(` in ~10 other test files (embed-db, security, reranker, embeddings, peek-cache, watcher, cli-parity, scope-completeness, meta-invariant, embed-pipeline). 1024 source `it(` → 1086 runtime is exactly this expansion, working as intended.
- **This is the identical finding the previous Mavis round raised as its "L-11"** ("invariant regex matches 1020 but runtime 1082") which we already determined working-as-intended; this round re-flagged it at HIGH and additionally mis-claimed the cause was "unidentified." Rated HIGH "per docs-claims methodology" — but a deliberate, internally-consistent metric is not an overclaim. **No change.** (Both numbers are honest; "1024 unit tests" = 1024 authored test cases.)
