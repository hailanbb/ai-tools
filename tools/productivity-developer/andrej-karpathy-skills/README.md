# 受 Karpathy 启发的 AI 智能体编码提效指南 (Karpathy Skills)

本工具由一系列专为 AI 编码智能体（如 Claude Code、Cursor 等）定制的原则与规约文件组成，旨在彻底改善 LLM 的开发行为，对抗错误假设、过度复杂化、无关代码编辑以及缺少测试验证等典型 AI 编码陷阱。

---

## 🛠️ 第一阶段：环境自检与首次初始化引导

### 1. 运行环境与依赖确认
本指南适用于支持项目级规则加载的 AI 客户端，主要为 `Claude Code` 与 `Cursor`。
- **依赖检测**：
  - 使用 Claude Code 插件方式前，请确认您已本地安装了 Claude CLI 并具有正常的 GitHub 登录权限。可运行以下命令验证：
    ```bash
    claude --version
    ```
  - 使用 Cursor 规则前，请确认您的项目工作区已在 Cursor 编辑器中打开。

### 2. 首次运行初始化与激活

您可以根据需要选择以下三种激活方案之一：

#### 🔌 方案 A：Claude Code 全局插件（推荐）
在 `claude` 终端交互会话中，依次运行以下命令安装：
1. 添加插件市场源：
   ```bash
   /plugin marketplace add forrestchang/andrej-karpathy-skills
   ```
2. 安装插件：
   ```bash
   /plugin install andrej-karpathy-skills@karpathy-skills
   ```
   *安装后，这些行为原则将在本地所有 Claude Code 会话中全局加载。*

#### 📄 方案 B：项目级 `CLAUDE.md` 注入（仅对当前项目生效）
如果只希望在特定项目里应用这套规范，可将规则注入到项目根目录的 `CLAUDE.md` 中：
- **新项目一键下载**：
  ```bash
  curl -o CLAUDE.md https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md
  ```
- **已有项目追加注入**：
  ```bash
  echo "" >> CLAUDE.md
  curl https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md >> CLAUDE.md
  ```

#### 🎯 方案 C：Cursor 专有项目规则（仅 Cursor 生效）
本仓库包含已配置好的 Cursor 项目规则。
1. 将搬运后的 `.cursor/rules/karpathy-guidelines.mdc` 规则文件复制到您当前开发项目的 `.cursor/rules/` 目录下。
2. Cursor 在后续的对话及代码生成中会自动激活并应用这些编码准则。关于更多细节，可以查阅本地的 [CURSOR.md](CURSOR.md) 说明。

## 🚀 第二阶段：核心执行工作流

本工具通过规约让 AI 代理在开发中遵循“少假设、极简化、精细修改、测试驱动”的闭环，以保障代码质量与 Diff 干净程度。

### 1. 核心四大开发原则

| 核心原则 | 解决的 AI 陷阱 | 达成效果 |
| :--- | :--- | :--- |
| **编码前思考** | 错误假设、隐藏困惑、缺少权衡 | 在编码前与人类确认意图，确保方向正确 |
| **简洁优先** | 过度设计、无谓抽象、堆砌冗余配置 | 用最少的代码和最直观的方式解决问题 |
| **精准修改** | 顺手改动无关格式、重构不该碰的区域 | 严格约束 Git Diff，仅编辑跟任务直接相关的代码 |
| **目标驱动执行** | 模糊交付、缺少闭环调试 | 以测试和明确的可验证指标指导生成与迭代 |

### 2. 各原则具体实施细则

#### 💡 编码前思考
- **明确陈述假设**：若对需求、已有 API 或运行逻辑有任何不确定，必须先向用户提问，拒绝闭门猜测。
- **呈现多种解释与权衡**：当需求存在歧义时，应列出不同实现方案的优劣供用户选择，而不是默默挑选一个。
- **敢于反驳与纠正**：若发现用户给出的架构或方法有逻辑漏洞，或存在极简的平替方案，必须以“定位弱点、推演代价、提供方案”的方式进行反驳。

#### 🧹 简洁优先
- **禁止过度工程**：不添加未要求的灵活性、配置项或额外抽象层。
- **严禁编写不必要的错误处理**：不为绝对不可能发生的场景编写冗余的处理逻辑。
- **保持精简**：如 50 行能搞定，绝对不写 200 行。
- **资深工程师评审视角**：自我评估“这套设计会被资深工程师认为臃肿吗？”若是，则立即简化。

#### 🎯 精准修改
- **不碰周边代码**：不以“顺手改进”为由修改相邻的代码段、注释、空行或格式。
- **绝对匹配现有风格**：即便自己有更好的偏好，也必须完美遵循原项目的缩进、命名和编程范式。
- **精准清理残留**：只清理因您这次编辑直接废弃掉的导入和变量，不删除预先存在的历史死代码。

#### 🔄 目标驱动执行
- **测试驱动开发 (TDD)**：
  - **修复 Bug 时**：必须先编写能够复现该 Bug 的失败测试，修复完毕后确保该测试通过。
  - **增加新逻辑时**：先编写针对边缘情况的测试，编写完逻辑后进行循环运行直至全部通过。
- **制定包含步骤与验证手段的计划**：对于多步复杂任务，先输出形如 `1. [步骤] -> 验证: [具体命令]` 的计划，并在执行中严格对照验证。

### 3. 自定义拓展与最佳实践
- **项目专属定制**：本指南更偏向于通用编码质量控制。如有特定架构规范（如“使用 TS 严格模式”），可在本指南下方的 `## 项目特定指南` 章节中继续追加。
- **灵活性调配**：本套规则倾向于**质量与严谨性**。在处理琐碎简单的单行拼写修复等微小任务时，可自行权衡跳过繁琐的验证流程，以保持开发速度。
